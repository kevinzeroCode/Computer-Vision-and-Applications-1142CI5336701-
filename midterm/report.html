3D Model Colorization from Multi-View Images

M11415015 | Computer Vision and Applications (CI5336701, 2026 Spring) | Midterm Project

1. Overview

This project colorizes the 3D point cloud Santa.xyz (95,406 vertices, each with XYZ position and normal) from seven calibrated photographs. The final pipeline is: (1) build a mesh for MeshLab picking, (2) manually select 3D / 2D correspondences, (3) estimate one 3x4 projection matrix for each image, (4) reject invisible or occluded vertices, and (5) blend visible colors into the final colored point cloud.

2. Point Selection Strategy

2.1 3D Control Points

The original point cloud does not contain triangles, so make_geometry_ply.py first converts it into Santa_geometry.ply for MeshLab annotation. Shared 3D landmarks were then picked on stable structural regions: hat tip, shoulders, welcome sign corners, feet, belt-side bumps, one back-belt point, and a small number of lower-body refinement points for trouser and shoe boundaries. The final shared list contains 35 candidate landmarks.

2.2 2D Correspondences and Practical Workflow

2D pixel coordinates were clicked manually with pick_2d_points.py. In practice, the same image was often clicked multiple times because some landmarks were easy to miss or place inaccurately on the first try. Instead of forcing a single annotation pass, multiple candidate point sets were tested and the best set was kept for projection estimation. This workflow matches the actual development process more reliably than assuming every manual click is correct on the first attempt.

Each image finally uses 9 to 13 correspondences, depending on viewpoint visibility. Front views use bilateral shoulders, welcome-sign corners, and feet; side or rear views use the visible shoulder, belt-side landmarks, the back-belt point, and selected lower-body boundary points when they improve alignment.

3. Projection Matrix Estimation

For each image, a projection matrix P was estimated from 3D / 2D correspondences by normalized DLT. Each pair contributes two linear equations, and the final homogeneous system was solved by SVD. To improve robustness, the implementation also evaluates solvePnPRansac with approximate intrinsics and keeps the lower-error solution. When a point set contains one unstable point, the code can drop the worst offender and keep the remaining correspondences if this lowers reprojection error.

4. Colorization with Visibility and Occlusion Handling

4.1 Vectorized Projection

All vertices are projected to every image in one vectorized NumPy pass. Each 3D vertex is converted to homogeneous coordinates, multiplied by P, and divided by depth to obtain image coordinates. Vertices with non-positive depth are rejected immediately because they lie behind the camera.

4.2 Back-face Culling and Z-buffer

For each camera, the view vector is computed from the estimated camera center to every vertex. If the dot product between the surface normal and the view vector is non-positive, the vertex is back-facing and excluded. To satisfy the project requirement on occlusion handling, a per-image z-buffer is then built by rasterizing the nearest camera-to-vertex distance into a downsampled depth map. A candidate vertex is accepted only if its distance is consistent with the stored nearest depth within a small tolerance. This step reduces color leakage from self-occluded regions and was especially important after the later lower-body refinements.

4.3 Color Sampling and Blending

Visible vertices are sampled by bilinear interpolation instead of nearest-neighbor lookup. To reduce blur caused by small reprojection mismatch between multiple views, the final implementation does not average every valid view equally. Instead, it scores visible views by a cosine-based confidence term, keeps the best two candidates for each vertex, and either selects the strongest view directly or blends the top two views only. This produced a sharper hat pattern and clearer eye details than the earlier all-view averaging version.

visible if: depth > 0, inside image, normal · view > 0, and dist_to_camera <= zbuffer + epsilon
sample color: bilinear interpolation on 4 neighboring pixels
blend rule: keep the top-2 scoring views, then choose best-only or weighted blend

5. Results

The final output files are Santa_colored.ply and Santa_colored.xyz. In the final configuration, 95,020 of 95,406 vertices receive valid color (99.6%), while 386 vertices fall back to neutral gray. The main visual improvement from the z-buffer and top-2 blending strategy is reduced color bleeding around the hat, eyes, trouser hem, and shoe boundaries. One practical lesson from the project is that lower-body boundary points are useful only when they are manually placed very carefully; inaccurate additions can easily degrade projection quality, so repeated point selection and validation are necessary.

6. Run Order

The submission folder is organized so the reviewer can follow the same sequence used in the project:

  • Step 0: run make_geometry_ply.py if a new MeshLab picking mesh is needed.
  • Step 1: run pick_2d_points.py to record 2D correspondences; if a point set is poor, click again and keep the better set.
  • Step 2: run estimate_projection.py to estimate the seven projection matrices and check reprojection errors.
  • Step 3: run colorize.py to produce Santa_colored.ply and Santa_colored.xyz.
  • Step 4: inspect the final model in MeshLab.

7. Conclusion

This project fulfills the required pipeline for manual multi-view colorization of a 3D point cloud. The final solution uses manual correspondence selection, normalized DLT-based projection estimation, reprojection-error screening, back-face rejection, z-buffer occlusion handling, bilinear sampling, and confidence-based multi-view blending. The result is a nearly complete colored model with improved local sharpness and more reliable visibility handling than the earlier non-z-buffer baseline.
